# 2026-06-17 Physics/Code Review of ALEPH_Cumulant

Reviewer setup: requested physics-specialized subagent plus local code review. The subagent (`Mencius`) independently reviewed the cumulant code and reported that the existing correlator self-test passed (`Correlator self-test passed`). This note combines that review with a local pass over the same code paths.

## Executive Summary

The core full-event and two-subevent cumulant algebra is largely correct for the implemented definitions. The code avoids self-correlations in the all-particle `<2k>` sums, uses ordered-particle denominators consistently, applies the standard `c2{4}`, `c2{6}`, `c2{8}` combinations, and recomputes derived observables inside the delete-one-chunk jackknife.

The main issues are not a simple algebra mistake in `v2{2k}`. They are physics/statistics interpretation issues: ratio errors in the eta-gap comparison ignore covariance, `v_n` jackknife errors are conditional when sign-invalid samples are dropped, thrust-axis results are binned by lab multiplicity, invalid values are encoded as zero, and the detector-level e+e- observable can be dominated by nonflow jet and momentum-conservation effects.

## Findings

### 1. Eta-gap ratio uncertainties ignore covariance

Severity: medium.

`src/CompareV22EtaGap.cpp:46` propagates ratio errors with the independent-error formula. The same helper is used for the lower-panel gap/inclusive ratio at `src/CompareV22EtaGap.cpp:102`, `src/CompareV22EtaGap.cpp:135`, and for CSV output at `src/CompareV22EtaGap.cpp:202`.

The inclusive `v2{2}` and eta-gap `v2{2, |Delta eta| > 2}` are measured from the same events and the same jackknife chunks. Their statistical fluctuations are correlated. Treating them as independent can overestimate or underestimate the gap/inclusive uncertainty.

Recommended fix: compute the gap/inclusive ratio separately for each leave-one-chunk sample and jackknife the ratio itself. A clean implementation would add ratio histograms in `AlephCumulantSummary.cpp`, because that file already has the chunk-level inputs and jackknife loop.

### 2. `v_n` jackknife errors skip sign-invalid leave-one samples

Severity: medium to high for high-order/thrust bins.

`src/AlephCumulantSummary.cpp:453` marks `hV2_*`, `hV2TwoSub_*`, and `hV224*` as histograms where zero values are treated as invalid. Leave-one samples with zero values are skipped at `src/AlephCumulantSummary.cpp:472`, and the jackknife variance is then computed from the remaining valid samples at `src/AlephCumulantSummary.cpp:497`.

This is reasonable for avoiding fake real-valued roots when a cumulant has the wrong sign, but the resulting error is conditional on only the sign-valid jackknife samples. Near a sign change, this is not a true delete-one estimate of the estimator. It can make high-order points look more stable than they are.

Recommended fix: if any leave-one sample has an invalid sign for a plotted `v_n`, either suppress/flag the central `v_n` point or quote the error on the raw cumulant `c_n{m}` instead. Another option is to store explicit validity masks or `NaN` values rather than using zero as the invalid sentinel.

### 3. Thrust-axis results are binned by lab-selected multiplicity

Severity: medium, mostly interpretation.

The lab multiplicity is defined as `summary.BeamPhi.size()` at `src/AlephChargedCumulants.cpp:508`. Thrust-axis tracks are selected separately using `pT` relative to the thrust axis at `src/AlephChargedCumulants.cpp:494`. However, both beam and thrust histograms are filled using `summary.LabMultiplicity` at `src/AlephChargedCumulants.cpp:1335`.

The cumulant denominators themselves use the actual number of selected tracks in the relevant axis frame, so the arithmetic is not internally inconsistent. The issue is the horizontal axis: thrust-axis points are plotted versus lab-selected charged multiplicity, not thrust-selected multiplicity.

Recommended fix: either keep this definition and label it explicitly as `lab selected Nch`, or add a mode to bin thrust-axis histograms by `summary.ThrustPhi.size()`. The existing `hSelectedThrustMultiplicity` helps diagnose the difference but does not affect the x-axis binning.

### 4. Invalid physics values are encoded as zero in summary and plotting

Severity: medium.

`src/AlephCumulantSummary.cpp:206`, `src/AlephCumulantSummary.cpp:214`, `src/AlephCumulantSummary.cpp:222`, and `src/AlephCumulantSummary.cpp:234` leave `v_n` bins at zero when the relevant cumulant sign does not allow a real-valued root. Plotting code then treats `(content,error) = (0,0)` as invalid, for example `src/CompareV22EtaGap.cpp:41` and `src/CompareTwoSubeventV2Multiplicity.cpp:46`.

This convention works for display, but it is ambiguous: a true zero-valued observable and an invalid/missing value are represented the same way. CSV tables can also contain zeros for invalid values, which can look like a real measurement unless the user knows the convention.

Recommended fix: use an explicit validity histogram, `NaN`, or a negative sentinel outside the physical plotting range. At minimum, CSV output should include a `valid` column for each plotted quantity.

### 5. Small-N thrust axis is approximate

Severity: low to medium.

For `momenta.size() <= 3`, `include/ThrustTools.h:36` tests only axes along individual particle directions. The exact thrust axis for a small set of vectors can lie along signed sums such as `p1 + p2`, not necessarily along a single particle.

This is probably a small effect for the default hadronic samples because the thrust input usually has many reconstructed particles, but it can matter for low-multiplicity events and for any `--ThrustChargedOnly 1` systematic run.

Recommended fix: use the same signed-sum candidate strategy for all multiplicities, or explicitly enumerate all nonzero signed sums for `N <= 3`.

### 6. Two-subevent `etaBoundary = 0` is not a finite eta-gap suppression

Severity: physics caveat.

The two-subevent production documented in the repository uses `--TwoSubeventEtaBoundary 0.0`; the parser default is also zero at `src/AlephChargedCumulants.cpp:1233`. The implementation splits tracks into `eta < -boundary` and `eta > boundary` at `src/AlephChargedCumulants.cpp:645`.

With boundary zero, the two subevents are opposite eta hemispheres but have no finite exclusion gap around eta = 0. This satisfies the requested `k` from A and `k` from B construction, but it should not be described as a large rapidity-gap nonflow suppression.

Recommended fix: present the current result as a two-hemisphere subevent cumulant. Add boundary variations such as 0.2, 0.5, and 1.0 if the goal is nonflow suppression.

### 7. Detector-level and e+e- physics caveats dominate interpretation

Severity: high for physics interpretation, not a code bug.

The note correctly states that the result is detector-level and not corrected at `AnalysisNote/main.tex:525`. There is no detector response correction, no acceptance correction, no tracking efficiency variation, no event-selection systematic, and no closure test showing that the method recovers a known generator-level truth observable.

In e+e- at LEP1, large `v2{2}` values can be produced by jets, back-to-back topology, momentum conservation, resonance decays, and local charge/fragmentation correlations. Calling the coefficient “flow” without qualification would be misleading. The eta-gap and two-subevent studies are useful diagnostics, but they do not by themselves remove all nonflow in a two-jet event topology.

Recommended fix: keep the language as detector-level angular-correlation/cumulant coefficients unless and until correction, closure, and systematic studies are complete. Add generator-level closure and variation studies before interpreting as collective flow.

### 8. Thrust-axis eta-gap is not the same object as a hadron-collider pseudorapidity gap

Severity: physics caveat.

For the thrust-axis result, `eta` and `phi` are computed relative to the event thrust axis at `src/AlephChargedCumulants.cpp:497`. A large `|Delta eta|` around the thrust axis has a different geometric meaning from a beam-axis pseudorapidity gap in heavy-ion or pp analyses. In two-jet e+e- events, it can preferentially compare particles in different regions of a jet or between hemispheres depending on the event topology.

Recommended fix: describe the gap as “pseudorapidity relative to the chosen axis” and include beam-axis vs thrust-axis comparisons as separate observables, not interchangeable gap definitions.

## Checks That Look Correct

- The full-event ordered correlator builder at `src/AlephChargedCumulants.cpp:523` uses a dynamic-programming elementary-symmetric construction. A particle can be selected once as a plus leg, once as a minus leg, or not selected; it cannot be selected twice. Multiplying by `half! * half!` and using `FallingFactorial(M, 2*half)` gives the ordered distinct-particle convention.
- The standard cumulant combinations in `src/AlephCumulantSummary.cpp:202` through `src/AlephCumulantSummary.cpp:235` are correct for `c2{2}`, `c2{4}`, `c2{6}`, `c2{8}` and the usual `v2{2k}` sign conventions.
- The eta-gap pair implementation at `src/AlephChargedCumulants.cpp:593` loops over unordered pairs and adds a factor of two to both numerator and denominator, which matches ordered-pair weights for `<2>`.
- The two-subevent implementation at `src/AlephChargedCumulants.cpp:633` uses elementary symmetric products in the two subevents and denominators `P(M_A,k) P(M_B,k)`, which is consistent with sampling `k` particles from each subevent.
- The summary code recomputes derived cumulants for every leave-one-chunk sample before assigning errors, rather than propagating errors only from raw numerator/denominator sums. This is the right structure for nonlinear cumulant observables.
